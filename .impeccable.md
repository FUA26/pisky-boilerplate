# Design Context - Zilpo

## Users

**Primary Audience**: Developers building SaaS products

**Context**: Technical users who want to skip boilerplate and get to shipping features quickly. They're evaluating Zilpo as their foundation - they need to trust it's well-built, understand how it works, and see themselves extending it.

**Jobs to be Done**:

- Kickstart a new SaaS project without weeks of setup
- Learn best practices for modern Next.js architecture from the codebase
- Ship a functional product fast with patterns that scale
- Focus on business logic, not infrastructure decisions

## Brand Personality

**Three Words**: Fast, opinionated, approachable

**Emotional Goals**:

- **Confidence** - This codebase knows what it's doing. The patterns are intentional.
- **Momentum** - No friction, no second-guessing. Just build and ship.
- **Warmth over coldness** - Feels like a capable friend guiding you, not a stern framework enforcing rules

**Voice**: Direct and decisive, but not rigid. "Here's the smart way to do it" not "Do it this way or else."

**Anti-patterns**: Academic, bureaucratic, timid, overly-flexible, enterprise-corporate

## Aesthetic Direction

**Visual Tone**: Softer and more approachable than the Linear-inspired baseline. Less "precision engineering tool," more "thoughtful workspace you enjoy spending time in."

**Theme**: Light mode default with dark mode support. Light feels more open and welcoming for daytime building sessions; dark for focused evening work.

**Color Strategy**:

- **Primary**: Teal/cyan at hue ~165 (`oklch(0.508 0.118 165.612)`) - energetic but not aggressive
- **Neutrals**: Currently pure grayscale - SHOULD be tinted toward brand hue for subconscious cohesion
- Move away from achromatic near-black → use warmer neutrals with subtle tinting
- Soften borders and edges → more generous radius (10px current), less harsh contrast
- Background surfaces should feel layered and inviting, not stark

**Typography**:

- **Body**: Inter Variable - Strategic choice for developer tools (technical, readable, excellent rendering)
- **Monospace**: Geist Mono - For code/terminal displays
- **Note**: Inter is on the reflex list but kept intentionally because it's RIGHT for this audience (developers expect and trust Inter for tooling)

**Spacing & Layout**:

- More whitespace → room to think
- Softer corners (8-12px range) - currently using 10px radius
- Fluid, breathing layouts → not rigid grids
- Visual hierarchy through scale and spacing, not just borders
- Asymmetric layouts with interesting compositions

**References**:

- **Keep**: Vercel's clean developer aesthetic, but warmer
- **Keep**: Notion's approachable workspace feel
- **Avoid**: GitHub's stark utilitarianism
- **Avoid**: Stripe's overly-polished marketing slickness

**Anti-references**: Stark black developer consoles, bootstrap dashboards, generic admin templates

## Design Principles

1. **Opinionated but not rigid** - Show the smart path clearly, but let devs deviate when needed
2. **Functional warmth** - Softness serves purpose: readability, comfort, reduced cognitive load
3. **Code-first aesthetics** - The product IS the code; design should showcase it, not compete with it
4. **Ship-ready patterns** - Every design decision should demonstrate "this is how you build it right"
5. **Approachable expertise** - Sophisticated implementation presented in a way that feels learnable, not intimidating

## Technical Notes

- **Component library**: shadcn/ui with custom theming
- **Styling**: Tailwind CSS v4 with OKLCH colors
- **Design tokens**: Centralized in `packages/ui/src/styles/globals.css`
- **Animations**: Using `cubic-bezier(0.16, 1, 0.3, 1)` for natural deceleration
- **Font loading**: Next.js font optimization with variable fonts
